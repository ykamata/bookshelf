# Project Agents.md Guide for OpenAI Codex (Nuxt 3 + Composition API + TypeScript)

This **Agents.md** file provides comprehensive, opinionated guidance for OpenAI Codex, GitHub Copilot, and other AI agents collaborating on this Nuxt 3 code-base.

## Tech Stack

* **Framework**: Nuxt 3 (LTS) — version ^3.11 or newer (Nitro v3)
* **Language**: TypeScript 5.x — with `strict: true`, `resolveJsonModule`, and path alias `~/*`
* **UI Layer**: Vue 3 + Composition API + `<script setup lang="ts">`
* **Styling**: Tailwind CSS v4 + UnoCSS (on-demand utilities)
* **State Management**: Pinia v3 (auto-imported stores)
* **Lint & Format**: ESLint v9 flat config via `@nuxt/eslint`, with Stylistic enabled
  — One-stop linting **and** formatting (**Prettier is not used**)
* **Testing**: Vitest + `@nuxt/test-utils` + Playwright for end-to-end testing
* **Continuous CI**: GitHub Actions (`node@20`, `npm ci`)

## Project Structure for OpenAI Codex Navigation

- `/app` — Global app configuration (`app.config.ts`), layouts, error views
- `/assets` — Static images & fonts (AI agents should not transform these)
- `/components` — Vue components; OpenAI Codex generates/edits `<PascalCase>.vue` files here
- `/composables` — Re-usable Composition-API logic (`use*` functions)
- `/pages` — File-system routes; Codex may scaffold new pages when requested
- `/plugins` — Nuxt plugins (client/server) that Codex can extend
- `/public` — Pass-through assets (Codex **must not** modify)
- `/server` — Nitro server routes / API endpoints
- `/stores` — Pinia stores (`*.ts`)
- `/tests` — Unit / E2E tests that Codex should maintain and extend
- Root files: `nuxt.config.ts`, `eslint.config.mjs`, `package.json`, etc.

## Coding Conventions for OpenAI Codex

### General Conventions for Agents.md Implementation

- **TypeScript 5** is mandatory for all new code generated by Codex
- Follow the existing ESLint Stylistic rules (flat-config via `@nuxt/eslint`)
- Respect `strict: true`; avoid `any`, prefer generics or utility types (`Pick`, `Omit`)
- Use meaningful variable & function names; add inline comments only for non-obvious logic
- Keep line length ≤ 100 chars and omit semicolons (Stylistic default)

### Vue 3 Components Guidelines for OpenAI Codex

- Use **functional single-file components** with the **Composition API** and `<script setup lang="ts">`
- Break large UIs into small, focused components
- Name files in **PascalCase.vue** and export emits/props with explicit typings
- Rely on Nuxt auto-imports (`ref`, `defineComponent`, etc.); do **not** import these manually
- Provide stable keys in `v-for`; avoid using the array index unless immutable
- Use `useAsyncData` / `useFetch` for remote calls; avoid raw `fetch` in components

### CSS / Styling Standards for OpenAI Codex

- Use **Tailwind CSS v4** utilities (with UnoCSS for on-demand generation)
- Follow the utility-first approach; fallback to scoped CSS only when necessary
- Prefer logical properties and responsive variants; avoid hard-coded pixel values

## Testing Requirements for OpenAI Codex

OpenAI Codex should run tests with the following commands (Vitest + Playwright):

```bash
# Run the complete test suite
npm run test

# Run a specific test file
npm run test -- tests/path/to/file.spec.ts

# Run tests with coverage
npm run test -- --coverage
```

> **Vitest** is configured via `vitest.config.ts`; Codex should create snapshot tests next to the component when appropriate.

## Pull Request Guidelines for OpenAI Codex

When OpenAI Codex assists in creating a PR, ensure it:

1. Includes a clear description of the change and its rationale
2. References any related issues or tickets
3. Passes **all** tests (`npm run test`) and programmatic checks (see below)
4. Provides screenshots or videos for any visual/UX changes
5. Focuses on a single concern; large refactors must be split into incremental PRs
6. Updates `CHANGELOG.md` following **Conventional Commits** when merging to `main`

All checks **must** pass before AI-generated code can be merged.

### Key Takeaways for AI Agents

- **ESLint (Stylistic) is the single source of truth** for both linting and formatting.
- Use TypeScript strictly; leverage generics, never fall back to `any`.
- Embrace small, well-typed Vue components and composables.
- Apply Tailwind utilities before reaching for custom CSS.
- Write tests alongside code; treat coverage as non-optional.
